# Skill: Progress Arbitration / 进度仲裁

监控项目执行进度，识别阻塞点，协调资源解决问题。

## Steps

### Step 1: Observe - 收集进度信息

```yaml
inputs:
  - task_list: "任务清单 JSON"
  - current_status: "各任务当前状态"
  - blockers: "已报告的阻塞问题"
  
actions:
  - 扫描各阶段完成情况
  - 收集角色反馈
  - 识别延期风险
```

### Step 2: Orient - 分析项目健康度

| 指标 | 绿 | 黄 | 红 |
|------|-----|-----|-----|
| 进度 | 按计划 | 延期 1-2 天 | 延期 3+ 天 |
| 阻塞 | 无 | 已识别但可控 | 严重阻塞 |
| 质量 | 测试全通过 | 少量失败 | 大量失败 |
| 资源 | 充足 | 紧张 | 不足 |

### Step 3: Decide - 仲裁决策

根据项目状态选择行动：

```yaml
decisions:
  on_blocker:
    - 分配资源解决
    - 调整任务优先级
    - 寻求外部支持
    
  on_delay:
    - 评估影响范围
    - 调整里程碑
    - 通知相关方
    
  on_quality_issue:
    - 增加测试资源
    - 触发代码审查
    - 安排技术支持
```

### Step 4: Act - 执行协调

1. 更新任务状态
2. 重新分配资源
3. 调整计划时间线
4. 生成进度报告

### Step 5: Validate - 确认决策有效

- 阻塞是否解除？
- 新计划是否可行？
- 相关方是否知悉？

## Output

### Progress_Report.md

```markdown
# 项目进度报告

## 概览
- **项目**：[项目名称]
- **报告日期**：2026-01-02
- **整体状态**：🟡 需关注

## 里程碑进度
| 里程碑 | 计划日期 | 状态 |
|--------|----------|------|
| M1 需求确认 | 01-02 | ✅ 完成 |
| M2 设计完成 | 01-03 | 🔄 进行中 |

## 阻塞问题
1. **[已解决]** API 设计需要更多输入
2. **[处理中]** 依赖库版本冲突

## 下一步行动
- 架构师完成 API 设计
- 开发准备开始编码

## 风险预警
- 无
```
